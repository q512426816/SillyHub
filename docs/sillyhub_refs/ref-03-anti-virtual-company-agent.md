# Ref 03：反“三省六部 / 虚拟公司式多 Agent”

## 文章核心观点

这篇文章反对把多 Agent 简单模拟成人类公司组织，例如：

```text
产品经理 Agent → 架构师 Agent → 开发 Agent → 测试 Agent → 运维 Agent
```

它认为这种“虚拟公司式多 Agent”看起来合理，但工程上容易出问题。

核心问题是：

- 人类组织分工是为了解决人的注意力、专业边界和协调成本。
- LLM 的瓶颈不是人类式分工，而是推理深度、上下文完整性和状态连续性。
- Agent 之间像部门一样交接文档，会导致信息在传递中衰减。
- 每一步都可能局部正确，但整体目标逐渐漂移。

## 讲得对的地方

### 1. 线性接力会导致上下文断裂

错误模式：

```text
A Agent 输出结论
  ↓
B Agent 只看 A 的结论
  ↓
C Agent 只看 B 的结论
  ↓
原始目标、推理过程、隐含假设逐步丢失
```

这对 AI 编码尤其危险，因为需求、架构、实现、测试之间高度耦合。

### 2. 多 Agent 的价值不是模拟部门

更合理的多 Agent 用法是：

```text
主线任务保持完整意图
  ↓
分派子 Agent 探索特定问题
  ↓
子 Agent 返回证据、风险、建议
  ↓
主线统一合并和推进
```

也就是：

> 推理链不能断，只能分叉再合并。

### 3. 外部状态比会话记忆可靠

Agent 不能靠聊天记录保持长任务连续性。应把关键状态外化到持久存储：

```text
spec.md：需求目标
plan.md：实施计划
progress.md：执行进展
decision.md：关键决策
review.md：审查意见
test-result.md：测试结果
diff.patch：代码变更
knowledge-candidates.md：候选知识
```

## 说得过头的地方

这篇文章容易让人误解为“职责分工 Agent 都不行”。这个结论过度了。

真正不行的是：

```text
Agent 之间线性接力、信息隔离、层层压缩、互相交权。
```

不是不允许职责分工。

SillyHub 仍然可以有：

```text
Planner：生成方案
Critic：对抗评审
Implementer：代码实现
Reviewer：代码审查
Tester：测试验证
Archiver：知识沉淀
```

但它们不能形成部门式接力，而必须由平台主线统一调度。

## 对 SillyHub 的启发

SillyHub 不应该做成“AI 虚拟公司”。

错误定位：

```text
虚拟软件公司：
产品部 Agent、架构部 Agent、开发部 Agent、测试部 Agent 层层传递。
```

推荐定位：

```text
主线任务驱动的委派式多 Agent 工程交付系统。
```

结构如下：

```text
用户需求
  ↓
Task Orchestrator 创建任务主线
  ↓
读取 spec / knowledge / project context
  ↓
主控保持完整意图
  ↓
按需委派：需求澄清、技术调研、方案评审、代码实现、测试生成、风险检查
  ↓
所有结果回流主线
  ↓
统一更新 plan / progress / diff / review
  ↓
测试通过后生成 PR
  ↓
归档知识
```

## 推荐写入设计文档的表述

```text
SillyHub 不采用“虚拟公司式多 Agent”架构，不将 Agent 简单模拟为产品经理、架构师、开发、测试等部门进行线性接力。

平台采用“主线任务 + 委派式 Agent”的协作模型。Task Orchestrator 始终持有完整任务目标、上下文、状态和权限边界；各 Agent 只作为特定阶段或特定问题的执行单元，被委派去完成分析、实现、验证或审查任务。

Agent 之间不直接进行权力交接，也不通过压缩后的结论文档层层传递任务。所有中间产物、关键推理、决策依据、代码变更、测试结果和审查意见都沉淀到外部状态文件与任务记录中，由主控流程统一读取、合并和推进。
```

## 一句话总结

> 多 Agent 不是虚拟公司，不是流水线接力，而是主任务下的分支探索、职责委派和结果回流。
