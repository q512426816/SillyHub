# Ref 02：Harness 不是目的，知识才是护城河

## 文章核心观点

这篇文章的核心观点是：Harness 工作流本身不是最终目的，团队和领域知识的沉淀才是长期护城河。

它提出：

- 工作流是可替换的，知识是可累积的。
- 没有知识沉淀的工作流是一次性的。
- 每次 AI 工程交付都应该沉淀业务规则、架构决策、坑点、流程经验。
- 知识应该分层、分类、标注成熟度，并有生命周期管理。
- 工作流应该服务于知识闭环：INIT 注入 → 执行中引用 → ARCHIVE 提取。

## 文章中可采纳的知识模型

### 知识层级

可参考五层：

```text
个人层：个人偏好、本地习惯
团队层：编码规范、提交规范、Review 标准
技术层：通用技术方案、组件经验、架构模式
业务层：行业规则、客户流程、领域模型
项目层：当前项目架构、接口说明、数据库结构、历史决策
```

### 知识类型

推荐保留五类：

```text
model：实体、字段、关系、数据结构
process：业务流程、审批流、状态机
decision：技术选型、架构决策及理由
guideline：推荐做法 / 禁止做法
pitfall：历史问题、风险、排查方式
```

### 知识成熟度

文章中的 `draft → verified → proven` 有参考价值，但 SillyHub 更适合扩展为：

```text
candidate：AI 自动提取，未确认
confirmed：人工审核通过
verified：在当前项目成功使用
promoted：跨项目复用，进入团队知识库
deprecated：过时或不再适用
```

## 对 SillyHub 的价值

SillyHub 不能只做“帮我写代码”。它应该让每一次交付都沉淀为团队资产。

任务闭环应包括：

```text
需求输入
  ↓
读取项目知识 / 团队规范 / 历史决策
  ↓
生成方案与任务计划
  ↓
编码、测试、Review
  ↓
提取候选知识
  ↓
审核、确认、沉淀
  ↓
下次任务自动复用
```

## 需要谨慎的地方

### 1. 不能全自动确权知识

AI 可以负责“发现候选知识”，但不能直接把候选知识写入正式库。

推荐流程：

```text
Agent 自动提取候选知识
  ↓
进入待审核区
  ↓
Reviewer Agent / 人工审核
  ↓
通过后进入项目知识库
  ↓
跨项目验证后提升到团队知识库
```

### 2. 成熟度不能只看引用次数

被多次引用不代表一定正确。成熟度提升还要看：

- 是否人工确认。
- 是否真实项目交付成功。
- 是否通过测试验证。
- 是否无线上事故。
- 是否有清晰适用范围和版本限制。

### 3. 自动衰减不能一刀切

某些知识虽然很少引用，但属于安全红线，不能自动降级。例如：

- 主分支禁止 Agent 直接 push。
- 生产数据库禁止直接修改。
- 高风险操作必须审批。
- 客户隔离和敏感数据规则。

### 4. 知识也需要权限

不是所有知识都能跨项目共享：

- A 客户业务规则不能随便注入 B 客户项目。
- 项目数据库结构不能被无权限用户查看。
- 个人偏好不能自动升级为团队规范。
- AI 提取的知识不能直接进入正式库。

## 向量数据库定位

SillyHub 知识库建议采用：

```text
Markdown / Git：知识正文和版本历史
关系型数据库：元数据、权限、成熟度、引用关系、生命周期
向量数据库：语义检索索引
```

关键原则：

> 向量数据库是检索索引，不是知识库本体。

检索流程应为：

```text
先按用户、项目、团队、客户、知识层级做权限过滤
  ↓
再按任务阶段、知识类型、成熟度做范围过滤
  ↓
最后使用关键词 + 标签 + 向量混合召回
  ↓
返回少量高质量知识注入 Agent 上下文
```

## 推荐写入设计文档的表述

```text
SillyHub 不应只定位为 Agent 执行平台，而应定位为 AI 工程交付与知识沉淀平台。
Harness 负责让 Agent 在受控流程中完成分析、设计、编码、测试和 Review；知识体系负责将每次交付中的业务规则、架构决策、技术坑点、流程经验和团队规范沉淀下来，形成可复用、可追溯、可演进的团队资产。
```
