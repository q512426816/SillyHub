前端错误处理规范化 — 设计原型

变更：2026-06-25-frontend-error-handling · author: qinyi · 2026-06-25T09:51:22+08:00

① 错误处理调用链（数据怎么流）

后端 AppError→ { code, message(中文), request_id, details }→ apiFetch→ 抛 ApiError
ApiError→ errMessage(err, fallback?) 新增·纯函数取中文文案
errMessage→ 按场景分发给 ↓
useNotify().error(err) 新增·hook | setError(errMessage(err)) | Modal.confirm(...)
errMessage 取文案规则：① code==="network_error" → 「网络连接失败，请检查网络后重试」（兜底浏览器英文）； ② 否则用 err.message（后端已中文，如 409「该 daemon 仍被 N 个 workspace 绑定…」）；③ 无 → fallback ?? "操作失败"。

② 展示策略决策树（什么场景用什么）

错误发生 ├─ 用户主动触发的操作（删除 / 创建 / 更新 / 启停） │ └─→ useNotify().error / .success antd toast 即时反馈 │ ├─ 页面加载 / 列表拉取 / 详情获取失败 │ └─→ setError(errMessage(err)) inline 红条 · 保留上下文 │ ├─ 表单字段校验错误 │ └─→ inline 字段错误 inline │ └─ 需要二次确认的危险操作 └─→ antd Modal.confirm(...) 非 window.confirm
铁律：任何路径都 不 把 err.code（英文 HTTP_xxx）拼给用户看 —— 消灭 D 反模式。

③ daemon runtime 删除交互流程（首个落地场景）

点「移除」→ Modal.confirm「确定移除运行时 X？…」
确认→ DELETE /api/daemon/runtimes/{id}→
204 成功→ notify.success("运行时已移除")→ 列表移除该卡片
409 被绑定→ notify.error(err) → toast 弹「该 daemon 仍被 N 个 workspace 绑定（myaaa），请先解除绑定」
404 / 403→ notify.error(err)→ toast 弹后端中文 message
修复前：409 → 500 白屏；修复后：409 → toast 中文提示，列表不动，用户知道要去 workspace 解绑。
