# 提案书（Proposal）— 工作台日历左点负载：过去实际 / 未来剩余

## 动机

个人工作台「工作日历」左点（负载）整月都按计划工时 `plan_task.work_load` 算：过去的日子也按计划显示、不能反映「实际干了多少」；未来的日子简单按计划平摊、不能反映「扣掉已用后还剩多少压力」。需要以「今天」为分界重算：过去看实际工时，今天及以后看剩余负载。右点（进度）现状已是纯计划，无需改动。

## 关键问题（现有方案为何不够）

1. **过去失真**：过去日期按计划工时显示，与实际已发生的负载脱节（实际可能多干/少干）
2. **未来失真**：未来日期简单按计划工时平摊，忽略已用掉的工时——剩 8 人天分到 11 天 vs 分到 20 天，压力完全不同，简单平摊看不出来
3. **两侧同源**：过去与未来都用计划工时，无法体现「计划 vs 实际」的偏差

## 变更范围

- `get_calendar()` 左点 `load_level`：过去用 `task_execute` 实际工时平摊（D-001~005）；未来用剩余负载 `(计划总量−已用)/剩余天数`（D-007）
- 补 / 改 `test_workbench_service.py`（过去/未来/已用扣减/跨月/兜底/右点回归）
- 前端 `work-calendar-panel.tsx` 图例加一行口径说明

## 不在范围内（显式清单）

- 不改右点 `alert_level`（`_task_alert`）任何逻辑
- 不改 `CalendarDay` schema / API 契约 / 前端组件结构
- 不改数据库 / 无 migration
- 不引入新抽象类（不走方案 B）
- 不改前端分档计算（不走方案 C）

## 成功标准（可验证）

- 过去日期按实际工时平摊显示负载（actual 区间覆盖即累计）
- 未来日期按 `(计划总量 − 已用) / 剩余天数 × 8h` 显示负载，用户原例（10 人天 1\~20 号、今天 10 号、已用 2 → 8/11 人天/天）可复现
- 过去无实际记录 → 灰点（`none`）；未来 `work_load` 空 / `end_time` 缺失 → 该任务不贡献
- 右点 `alert_level` 行为零回归
- 前端图例说明「过去按实际 · 今天及以后按剩余」
- backend workbench 单测全绿、ruff/mypy 通过
