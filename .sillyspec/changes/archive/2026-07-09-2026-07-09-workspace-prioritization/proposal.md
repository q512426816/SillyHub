# 提案书（Proposal）— 工作区前置化

## 动机

"工作区 + 守护进程（daemon）"是平台绝大多数功能的前置依赖（约 20 个菜单未进工作区时灰显不可点），但在当前 UI 里，工作区只是与功能菜单平级的一个普通菜单项，没有体现"顶层会话"地位。用户登录后无法直观地"先进入工作区建立上下文（含确认守护进程），再用依赖工作区的功能"。

## 关键问题

1. **工作区定位低、无全局切换器**：与功能菜单平级，切换工作区需退回列表页点卡片，顶栏（`top-bar.tsx`）不显示当前工作区。
2. **上下文无缓存层**：当前工作区靠 `useWorkspaceId()` 正则解析 URL，顶栏无法反映上下文（连"当前在哪个工作区"都不显示）。
3. **daemon 绑定是事后补救**：进工作区详情页后才由 `WorkspaceBindingGuard` 弹引导，守护进程不是和工作区一起建立的"前置会话"。
4. **落地页是双入口标题页**：登录后落到 `app/page.tsx` 标题页，不是"选工作区"的第一步。

## 变更范围

- **顶栏加全局工作区切换器**：显示当前工作区名 + daemon 在线状态徽标，下拉切换跳到新工作区同模块。
- **登录后强制先选工作区**：客户端守卫（dashboard layout 层），未选时侧边栏只露全局菜单，选择器页常驻平台后台旁路入口（不困住管理员）。
- **落地页改工作区选择器**：登录后直接进选择器，取代双入口标题页。
- **daemon 绑定在选择器/切换器内完成**：未绑定的工作区点击弹绑定弹窗，绑好才进。
- **URL 路径派生保持真相源**：新建 `stores/workspace.ts` 仅缓存（非 persist），深链/刷新天然工作。

## 不在范围内（显式清单）

- 不改后端表结构 / API（只读消费现有 `my-binding` 接口 + daemon 实例在线状态）
- 不引入 Next.js middleware（token 在 localStorage，middleware 读不到，改动过大）
- 不改 `menu-permissions.ts` 菜单数据结构（继续用 `absolute` 标记区分平台级/工作区级）
- 不做工作区级路由组重组（不移动 `/workspaces/[id]/*` 路由文件）
- 不做移动端响应式
- 不改 daemon 生命周期 / 会话 / lease 逻辑

## 成功标准（可验证）

- AC-1：登录后落到 `/workspaces` 选择器，未选工作区时侧边栏只露全局菜单（平台管理/系统设置/PPM）
- AC-2：选择器页有"平台管理 / 系统设置"后台旁路入口，可直接进不报错
- AC-3：顶栏切换器显示当前工作区名 + daemon 在线状态徽标
- AC-4：在工作区 A 的「变更」页用切换器切到 B，跳到 B 的「变更」页（保留模块段）
- AC-5：未绑定 daemon 的工作区点击弹绑定弹窗，绑好才进
- AC-6：daemon 离线时切换器标红，不阻断进入（依赖功能由各页自判）
- AC-7：刷新 / 深链 `/workspaces/B/changes` 直接进入，不丢上下文
- 现有平台后台路由（`/admin` `/settings` `/ppm` `/runtimes`）零回归
- 现有菜单灰显逻辑、详情页 `WorkspaceBindingGuard` 编辑入口保留
