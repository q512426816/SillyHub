# 提案书（Proposal）— 里程碑明细提交自动创建任务计划

## 动机
里程碑明细（项目计划的叶子节点）变成「完成」后，理应产生一条可执行、可统计工时、能上看板的「任务计划」。但当前代码里明细走完状态机只写一条流程履历，**不会**自动生成任务（`PlanTask`）。用户每次都要再去任务页手动重建一条并挂回明细，既繁琐又容易漏建、错挂，导致「项目计划」与「实际任务台账」脱节。

## 关键问题
1. **明细与任务脱节**：明细 `done` 后无任何机制生成任务，`PlanTask.ps_plan_node_detail_id` 软关联字段形同虚设（仅迁移脚本/测试/手动建任务时才被赋值）。
2. **重复与孤儿风险未定义**：即便要补这个联动，明细还存在「编辑、变更（版本链）、删除」等生命周期，若无统一规则，极易产生重复任务或指向已删明细的孤儿任务。
3. **事务一致性无保障**：若联动简单堆在明细操作之后，明细提交成功但建任务失败会造成「明细显示完成、任务页却找不到」的不一致。

## 变更范围
在 `plan/service.py` 内聚实现「明细-任务联动」：明细变完成（手动提交 / Excel 导入）时自动建一条 `PlanTask` 挂在执行人名下，字段从明细映射、项目信息回溯自带；明细编辑、变更、删除时同步更新 / 迁移 / 解关联这条任务。全程在明细操作的同一数据库事务内完成（强一致），版本链查重防重复。

## 不在范围内（显式清单）
- 不补建历史已 `done` 的明细任务（仅对新提交/导入生效）。
- 不改明细状态机（`draft→done` 保持现状）。
- 不改 `PlanTask` 表结构（复用现有字段与 `ps_plan_node_detail_id`）。
- 不新增前端页面（复用现有 milestone-details 提交入口与 task-plan 任务页）。
- 不接管任务执行流（建出任务初始「未开始」，执行/工时/看板仍由 task/kanban 子域独立维护）。
- 不引入异步/事件总线（同步同事务联动）。

## 成功标准（可验证）
- 手动提交一条明细（执行人已填）→ 任务页/看板立刻出现一条归属该执行人的任务，字段与明细一致。
- Excel 导入 N 条完成明细 → 自动生成 N 条任务；任一失败则整批导入回滚。
- 编辑已完成明细的执行人/工期/主题 → 对应任务字段同步，任务自身状态不被重置。
- 明细发起变更 → 任务迁移到新版本，不产生第二条任务。
- 删除明细 → 任务保留但解除关联（无孤儿）。
- 联动任一步失败 → 明细操作整体回滚，无半成品数据。
- 既有明细 CRUD 路径行为不变；移除联动调用即可完全回退。
