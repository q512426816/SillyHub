# 提案书（Proposal）— 工作台日历负载修正 + 实际执行流程重设计

## 动机

个人工作台 `/ppm/workbench` 工作日历过去日期左点负载标黄错误：账号 180024（覃艺）2025-07 某天实际执行「1人天 + 0人天」本应饱和标绿却标黄。根因是后端把每条实际执行工时按起止天数**平均平摊**（跨多天记录稀释 <8h 标黄），而详情面板按覆盖日显示**原始整条**工时，两套口径打架。

用户在排查中进一步明确：不只是改负载算法，而是要**重设计整个实际执行流程**——支持启动/提交/完成三态、多次填报、禁止跨天（跨天前端拆分），问题处置同步，执行记录在详情可见。

## 关键问题

1. **负载标黄**：过去侧平摊稀释 + 详情口径不一致 → 用户困惑（"详情写1人天却标黄"）。
2. **执行流程缺失**：当前单按钮+checkbox，无独立启动、无提交/完成双按钮、不支持多次填报、无跨天约束。
3. **新录入数据无 actual 区间**：ExecuteTaskDialog 不填起止时间 → TaskExecute actual 空 → 即使改求和，新录入记录日历仍不显示（D-005 强制回填解决）。
4. **problem 处置无执行记录**：done_task 只追加 handle_info，不产生 TaskExecute（D-007 解决）。

## 范围

- task-plans 执行流程重设计（启动/执行弹窗提交完成双按钮/多次填报 1:N/跨天前端拆分）
- 后端 execute_plan 状态机改造（action 替代 submit）+ actual 起止强制写入 + 跨天校验
- problem-list 处置镜像（done_task 创建 TaskExecute）
- workbench 过去侧负载求和（推翻平摊）
- 执行记录详情可见（task 复用端点 / problem 扩 problem_task_id 过滤）
- 跨前后端 ~12 文件，5 Wave

## 不在范围内（Non-Goals）

- workbench 未来侧负载（剩余负载平摊保留）
- 右点 alert_level（07-15 已落地）
- 历史跨天迁移数据清理（规则11）
- 表结构变更（字段已齐）
- 问题变更（change）链路
- 批量 execute 端点（YAGNI，前端循环单条）

## 成功标准

1. 账号 180024 工作日历过去日期：当天实际执行工时求和 ≥8h 标饱和（绿），<8h 标有空余（黄）——「1人天+0人天」标绿。
2. task-plans：未开始任务可「启动」进入进行中；进行中可「执行」弹窗，提交（回未开始）+ 完成（转已完成）双按钮可用；支持多次填报产生多条执行记录。
3. 跨天：前端检测跨天拆分多天单独填；后端对跨天执行记录返回 422 拒绝。
4. problem-list 处置：提交/完成均产生执行记录，详情可见。
5. task-plans/problem-list 列表详情可查历次执行记录（开始/结束/耗时/说明/结果）。
6. 新录入的执行记录在日历过去侧正常显示（actual 区间非空）。
7. backend pytest + frontend vitest 全绿（含新增 test_router.py）。
