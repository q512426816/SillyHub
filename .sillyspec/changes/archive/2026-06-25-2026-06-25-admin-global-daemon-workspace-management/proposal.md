# Proposal

## 动机

平台管理员需要作为平台级运维角色查看并管理所有用户的守护进程运行时和工作区。当前运行时页面主要围绕当前用户数据，工作区列表虽已有平台管理员全量分支，但缺少人员维度筛选、名称/类型/状态筛选、真实分页和别名定义。资源数量增长后，卡片列表难以定位，也无法清楚表达“这是哪个人的哪个资源”。

## 关键问题

1. 现有 daemon runtime 列表按当前用户查询，平台管理员无法稳定看到全部用户的 runtime，也无法跨 owner 做启用、禁用、删除等卡片管理动作。
2. 工作区与守护进程卡片缺少服务端搜索和分页。前端拉全量再过滤不适合资源增长，也无法实现人员过滤的权限语义。
3. 资源原始名称来自 daemon 注册或 workspace 扫描，不适合直接拿来做可随时调整的展示别名。需要独立别名字段，既方便人识别，又不破坏原始语义。

## 变更范围

- daemon runtime 增加 `display_alias`，新增分页查询端点和别名 PATCH，平台管理员可跨 owner 管理 runtime。
- workspace 增加 `display_alias`，扩展既有列表查询参数和 PATCH DTO。
- 两类列表返回 owner 展示信息，平台管理员可按人员过滤。
- `/runtimes` 和 `/workspaces` 页面增加筛选条、分页、别名编辑和统一卡片样式。
- 增加后端测试与前端类型/交互验证，覆盖权限隔离、筛选分页、别名更新和路由顺序。

## 不在范围内（显式清单）

- 不改变 daemon 注册、heartbeat、lease、session 生命周期。
- 不让平台管理员接管或查看其他用户的交互式 session 内容。
- 不引入通用 resource_aliases 表或个人化别名。
- 不重做 admin 用户中心、角色中心或菜单权限体系。
- 不为历史资源自动生成默认别名。

## 成功标准（可验证）

- 平台管理员调用新分页 API 可看到全部用户的 runtime / workspace，并可按人员过滤。
- 普通账号传入 `user_id` 不能扩大可见范围。
- 两类资源都可更新 `display_alias`，卡片标题优先展示别名，空别名回退原始名称。
- `GET /api/daemon/runtimes` 仍返回数组；新分页能力走 `GET /api/daemon/runtimes/page`。
- `GET /api/daemon/runtimes/page` 不被 `{runtime_id}` 动态路由抢先匹配。
- 两页有服务端分页器、筛选条、别名编辑入口，且样式符合既有系统风格。
